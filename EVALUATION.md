# Deep Evaluation · 深入评价

**AgentWorld v1 vs v2 — 两个项目，两种哲学**

---

## AgentWorld v1 — 图优先的生产级仿真引擎

### 做得出色的地方

1. **拓扑-内容解耦是真正的创新。** 引擎内核只操作抽象的 `type_id` 整数和 `entity_id` 字符串——核心逻辑中从不引用实体名称或类型。`domain.json` 和 `node_config.json` 从外部注入所有语义。这意味着替换这两个文件就可以创建一个全新的世界，**Python 代码零改动**。我在任何其他 LLM-agent 框架中都没有见过这种程度的领域无关性。

2. **图作为单一真相来源。** 每个实体是一个节点；每个关系是一条带数量的边。库存不是一个单独的表——它就是 npc→item 的边。区域归属就是 npc→zone 的边。"库存"和"图"之间不可能出现不一致，因为它们就是同一个东西。`supply_view()` 的递归聚合和 `build_components()` 的 BFS 遍历是对图论的优雅运用，服务于实际的仿真需求。

3. **度守恒校验（Σ=0）是一个聪明的约束。** 借鉴热力学思想，区分内部转移（必须守恒）和系统边界流动（消耗、采集——可以不守恒）。标记为 `is_conserved: true` 的物品（金币、草药、武器）参与校验；来自环境的可消耗物品不参与。基于分组的守恒意味着一次交易的不平衡不会污染另一次交易。这是经过深思熟虑的设计。

4. **带反馈重试的校验注册表。** 8 项检查系统，每层通过掩码激活，架构干净。重试循环从 `CheckFailure` 对象构建结构化反馈（包含错误码、修复提示，以及 LLM 上一次的原始输出），这正是 LLM 输出修正应该做的方式——给它确切的失败 JSON 并告诉它哪里错了，而不是说"再试一次"。

5. **生产级的批量执行。** `run_50ticks.py` 将每个 tick 作为独立子进程运行，20 分钟超时，2 次重试，幂等跳过检查（检查已有 REPORT.md），以及基于正则的 stdout 指标提取——这是经过实战检验的批量处理。基于 contextvars 的并行追踪插桩也很精巧。

### 不足的地方

1. **全局 tick 锁步扼杀了涌现性。** 每个 tick 所有 13 个 NPC 同时行动。一个 NPC 无法在同一 tick 内对另一个 NPC 的动作做出反应——反应总是滞后一个 tick。这从根本上限制了因果链仿真：如果杰洛特进入酒馆，丹德里恩不能在同一刻向他打招呼，必须等到下一个 tick。世界感觉像是一连串静止帧，而不是流动的生命。

2. **LLM 输出是僵化的结构化操作。** LLM 必须输出 `{"op":"delta","src":"npc_97845b74","tgt":"item_coin","delta":-5}`——一个机器可读的图操作。从引擎角度看很优雅，但从 LLM 角度看很受限。LLM 无法表达"讨价还价"或"因为老板睡着而不付钱就走"这样的细微差别。一切都缩减为数量变化。

3. **图边的语义过载。** 边类型携带了硬编码在 Python 中的隐式含义（`npc_zone`、`npc_item`、`zone_zone` 等）。添加新的关系类型（如 `npc_owes_debt_to`）需要在多处修改代码。图引擎对节点是领域无关的，但对边是领域感知的。

4. **Prompt 组装分散在代码和配置之间。** 适配器有 30 多个 `slot_*` 方法，格式化来自 `domain.json` 的模板，但槽位定义及其排序存在于 `adapter.py` 中。添加新的 prompt 需要同时修改 Python（定义模板）和 JSON（定义槽位内容）。它不是纯粹由配置驱动的。

5. **NPC 没有内部驱动力。** 行为完全由 prompt 驱动：LLM #1 收到一个槽位注入的 prompt，包含 npc_state、周围环境和决策指引，然后输出计划。NPC 不会"感到口渴"——它只是被告知了 vitality/satiety/mood 的数字，并被期望做出计划。这效率很高，但缺乏能创造意外行为的自底向上驱动力模型。

## AgentWorld Async (v2) — 分层架构的自治沙盒

### 做得出色的地方

1. **位置即关系，哲学上干净。** 通过消除所有父子/容器/从属语义，使空间共置成为唯一隐式关系，架构消除了一整类耦合。吧台上的一杯酒只是两个同在 [7,2] 的实体——没有 `owner_id`，没有 `parent_entity`，没有 `inventory` 映射。把酒移动到 [3,5]，它就在"桌子上"了，代码零改动。这在哲学上是优雅的。

2. **带统一接口的分层架构。** 每层暴露一个方法（`see()`、`interact()`、`hear()`）——代码定义签名，YAML 定义返回值。这意味着新增实体类型或动作需要**零 Python 改动**。InteractionLayer 的动作名（"饮用"、"搭话"、"倚靠"）是来自 YAML 的纯字符串，不是 Python 方法名。LLM 看到的是人类可读的动作列表，而不是方法签名。

3. **LLM 作为裁判而非图操作生成器。** 裁判（LLM #2）看到双方的私有属性 + 周边实体，然后输出叙事 + 属性变化。这对 LLM 来说比结构化图操作更自然。它允许细微的结果："偷喝"（喝了不付钱）导致 `{thirst: -20, bar_counter.mood: -8}`——LLM 理解了社会语境，而不仅仅是图的数量变化。

4. **通过 inbox 实现真正的 agent 自治。** Agent 之间的交互走异步 inbox 消息，不走裁判。每个 agent 自己的 LLM 决定交互如何影响其属性。这防止了"你的属性被我外部裁定"的问题——在 v1 中，裁判的输出可能同时改变双方。在 v2 中，你自己控制你的心情变化。

5. **带 busy/idle 状态的混合异步。** busy-queue 模型（agent 在行动期间是"忙碌"的，但仍能接收和回复 inbox 消息）镜像了现实行为：你不能在喝酒中途走开，但可以对跟你打招呼的人点点头。

6. **纯 YAML 配置。** 所有 prompt 文本、槽位定义、动作规则、实体属性、区域定义和瓦片地图都存在于 YAML 中。Python 代码**零行硬编码 prompt 字符串**。这是配置驱动设计的北极星。

7. **三模式交互分发。** `resolve=rule`（引擎计算）、`resolve=llm`（LLM 裁判）、`target_type=agent`（inbox）。全来自一个 YAML 字段。InteractionSystem 分发到正确的代码路径，调用者无需知道使用了哪种模式。

### 不足的地方

1. **没有正式的校验系统。** v1 的 8 项检查注册表（含度守恒验证）在 v2 中缺失。LLM #2 裁判被期望产出有效输出，但没有 `capacity_upper_bound` 检查，没有 `entity_existence` 验证，没有度守恒。属性夹紧存在但纯粹是数值性的——它无法检测语义错误，比如"LLM 收了 5 枚硬币但 agent 只有 3 枚"。

2. **没有关系图模型。** 位置即关系对局部交互有效，但无法表达持久的非空间关系。如果杰洛特"欠丹德里恩一个人情"，除非写进 `private_attrs` 注释，否则无法表示——而 LLM 可能不会一致地读取。v1 的带语义类型的图边（`npc_debt_to`）可以自然地处理这种情况。

3. **规模是 demo 级别。** 一个 agent，一个 zone，五个实体，五次交互。架构展现了前景，但尚未在规模下测试：10+ agent 的交互半径重叠，50+ 并发 LLM 调用，持续多 agent 负载下的内存/性能。

4. **EventEntity 生命周期未经实战检验。** 动态事件生成和自动过期是个聪明的想法，但尚未在并发的观察者、快速生成/销毁周期、或事件实体占据相同坐标等边缘情况下测试过。

5. **没有持久化层。** demo 在内存中运行。v1 的统一 `nodes` 表（带 upsert 语义、自种子、`sync_graph_to_nodes()`）是一个经过验证的持久化模型，v2 尚未采用。

6. **外部 API 尚年轻。** REST + WebSocket 层对基本的移动/交互/感知操作有效，但尚不支持流式 LLM 输出、处理 agent 断开/重连、或对同一实体的并发访问。

## 综合评价

这两个项目代表了**根本不同的哲学**：

**v1 是为研究者和世界设计者打造的仿真引擎。** 它优化的是正确性、确定性和可复现性（批量运行 + REPORT.md），以及领域可移植性（换两个 JSON 文件 = 新世界）。其图模型和度守恒验证器具有数学上的严谨性。其 tick 同步模型适合研究 50+ tick 的系统级动力学。代码库厚重但自律——1400 行的适配器、1100 行的图引擎、contextvars 插桩、子进程隔离。这是面向特定用例的生产代码：大规模运行和分析猎魔人世界仿真。

**v2 是为建造者和修补者打造的 agent 自治沙盒。** 它优化的是表现力（LLM 叙事而非图操作）、可扩展性（YAML 中新增实体/动作）和实时交互（异步循环、WebSocket）。其分层架构和位置即关系使得原型构建新世界变得极其简单。混合异步模型（busy-queue + inbox）在概念上比纯回合制或纯异步更丰富。但它缺乏 v1 的严谨性——没有正式校验、没有图模型、没有持久化。代码库精简——约 30 个文件，平均每文件约 50 行——但带有原型的粗糙感。

**如果我要为生产环境构建多 agent 平台：** 我会取 v1 的图引擎 + 校验注册表 + 持久化，与 v2 的分层架构 + YAML 配置 + agent 自治 + 混合异步合并。图提供严谨的关系建模；层提供干净的关注点分离；YAML 配置提供可扩展性；异步提供响应性。单独任何一个项目都不是完整的答案——但两者一起描绘了一个真正有能力的多智能体仿真引擎的架构。

**如果我是 LLM-agent 研究者：** v1 是更有趣的代码库。它的拓扑-内容解耦、度守恒、和带反馈重试的 8 项校验注册表是大多数 LLM-agent 框架缺乏的新颖贡献。v2 的理念（位置即关系、分层架构、LLM 裁判）在架构上具有创新性，但需要 v1 的严谨性才能成为生产就绪的产品。
